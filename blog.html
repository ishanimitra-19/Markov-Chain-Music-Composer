What I Learned Building a Markov Chain Music Composer from Scratch

Lab 5 — Automated Composition  |  Web Audio

When you first hear about Markov chains for music generation, the concept sounds almost absurdly simple: count which notes follow which other notes in a piece, then use those counts as probabilities to pick what comes next. No neural networks, no gradient descent, no 200-page papers. Just a table of counts and a random number generator.

I assumed this simplicity meant the results would be obviously bad — rhythmically incoherent, harmonically random noise. What surprised me is that even a first-order chain over a handful of bars produces something that sounds vaguely like the original melody. Not good music, exactly, but clearly descended from it. The experience of hearing that for the first time made me think differently about what "musical structure" actually is.

How a Markov chain actually works here

The chain learns from a sequence of note names — C4 C4 G4 G4 A4 A4 G4 ... — and builds a transition table. For an order-1 chain, each state is a single note. For an order-2 chain, each state is a pair of consecutive notes. The chain records how often each state is followed by each possible next note across the training data. 

// Simplified: what buildChain produces for order=1
// from "C4 G4 A4 G4 F4"
{
  "C4": { "G4": 1 },
  "G4": { "A4": 1, "F4": 1 },
  "A4": { "G4": 1 }
}

Generation then works by picking a random starting state, sampling the next note proportionally to the counts, shifting the window forward, and repeating. If you reach a state you haven't seen during training (which can happen at higher orders if training data is sparse), you jump to a random known state and keep going. This dead-end recovery is a small detail that makes a real difference in practice.

The order dial: the most interesting knob

The single most illuminating thing I built was the order slider. Here is what the extremes sound like and what they tell you:

Order 1 — each note depends only on the previous note. The output is melodically plausible (notes that co-occur in the training data tend to recur together) but structurally formless. It wanders. You get something that sounds vaguely "in the style of" the original, the way a parody sounds like the thing it's parodying.

Order 4 — each note depends on the previous four. With a short training melody like Twinkle Twinkle, almost every 4-gram is unique, so the generated sequence rarely diverges from the original. You're essentially playing back training data with a few random jumps.

Higher order = more faithful, less creative. Lower order = more creative, less coherent. This tradeoff is not a bug — it's the fundamental tension in any generative model.

What I didn't expect is that this tradeoff is so sharp. There's no obvious "sweet spot" — the right order depends entirely on how much training data you have and what kind of output you want. With just 64 notes of Twinkle Twinkle, order 2 or 3 gives the best balance. Feed it a full Bach chorale and order 3 or 4 becomes interesting.

Rests are notes too

I included R (rest) as a first-class token in the note vocabulary rather than filtering it out. This turned out to be more important than I expected. Rests carry rhythmic information — they appear at phrase boundaries in Twinkle Twinkle, and the chain learns this. An order-2 chain quickly figures out that rests tend to follow D4 or G4 in that melody. If you strip rests, you lose the phrase structure entirely and everything collapses into an undifferentiated stream.

The transition matrix as a diagnostic

The matrix visualization (rows = states, columns = next notes, cell = probability) started as a debugging tool and became the most pedagogically useful part of the app. You can immediately see the structure of the training data:

At order 1, most states have several outgoing transitions with real probability mass spread across them — the melody reuses notes in varied contexts. At order 4, the matrix is very sparse (most states appear exactly once) and the transition probabilities are all 1.0 — each long context uniquely determines the next note. Sparsity in the matrix is a visual warning that your order is too high for your training data.

WebAudio: one oscillator per note

Each note spawns its own OscillatorNode and GainNode, connected through a short attack and exponential decay envelope. Oscillators in WebAudio are cheap and designed to be created and discarded — you should not try to reuse them. The pattern is: 

const osc  = ctx.createOscillator();
const gain = ctx.createGain();
osc.connect(gain);
gain.connect(ctx.destination);
// schedule envelope
osc.start(ctx.currentTime);
osc.stop(ctx.currentTime + duration);
// osc is garbage collected after stop()

One counterintuitive thing: you must not call audioCtx.close() immediately after osc.stop() — the context needs to stay alive until the scheduled audio finishes playing. Closing it early cuts off the last note. The fix is to close the context only when the user explicitly stops playback or the full sequence ends.

What this taught me about music

Building this made me think about melody in a new way. A melody is not just a sequence of pitches — it is a probability distribution over "what note comes next given the recent context." The Markov chain makes that distribution explicit and manipulable. When you hear a melody and recognize it as belonging to a certain style or genre, you are in some sense recognizing the shape of its transition distribution.

The failure modes of the chain — aimless wandering at order 1, faithful reproduction at order 4 — bracket the interesting middle ground where real composition lives. Composers internalize the transition statistics of their musical tradition and then make choices that are locally probable but globally surprising. The Markov chain can only do the locally probable part. That's a useful thing to know.

Going further

The natural extensions from here are: learning from MIDI files instead of hardcoded note strings (the blobToNoteSequence function from the Magenta library handles this), modeling rhythm separately from pitch (the current implementation treats every note as the same duration), and allowing the user to constrain generation to a particular key or scale. Each of these is a relatively small addition, but the third one — scale constraints — is interesting because it amounts to layering music theory on top of the statistical model rather than inside it.

The code for this project is in the same directory as this post. The composer runs entirely in the browser with no server or external dependencies — just three plain JavaScript files and a stylesheet.
